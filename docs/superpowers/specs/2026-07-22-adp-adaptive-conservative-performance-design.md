# ADP Adaptive Conservative Performance Optimization Design

## Статус

Одобрено пользователем 2026-07-22. Документ описывает стратегию максимального
ускорения активного lowercase `adp/` при контролируемом приближении и
консервативном ограничении потери качества.

## Контекст

Профилирование показывает два разных режима нагрузки:

- в запусках с `d=100`, `n=1000`, `J=1000`, `P=32` и временем около
  `8.02 s` построение локальных статистик занимает около `5.53 s` (`69%`),
  оптимизация — около `1.70 s`, из них beta CG — около `1.50 s`;
- в текущих тяжёлых запусках алгоритм часто исчерпывает все `4 * 20 = 80`
  внутренних шагов, поэтому beta CG становится главным bottleneck.

Следовательно, оптимизация только одного этапа не максимизирует итоговый
выигрыш. Стратегия должна одновременно уменьшать стоимость статистик,
стоимость одного beta solve и количество solve-вызовов.

## Цель

Добавить opt-in режим `adaptive_conservative`, который сокращает реальный
вычислительный бюджет, но автоматически повышает его при недостаточной
надёжности. Исходный строгий режим остаётся доступным как reference и
безусловный fallback.

Целевые performance gates:

- не менее `1.5x` для точного ядра статистик;
- медиана не более `4 s` для reference-конфигурации с исходным временем
  около `8.02 s`;
- не менее `2x` по медиане для CG-heavy задач;
- не менее `1.5x` по p90 больших задач;
- не более `5%` регрессии на маленьких задачах;
- не более `10%` роста peak RSS.

Это критерии принятия, а не заранее гарантированные результаты. Каждый из них
должен быть подтверждён повторяемым before/after benchmark.

## Ограничение качества

Кандидат сравнивается со строгим режимом попарно на одинаковых `X`, `y`,
центрах, направлениях и seed. Для каждой успешной пары обязательны:

- `cosine_abs_strict - cosine_abs_candidate <= 0.005`;
- относительное ухудшение полного objective не более `1%`;
- отсутствие новых `numerical_failure`;
- отсутствие роста числа `nonconverged`;
- конечный, детерминированно воспроизводимый результат.

При нарушении любого ограничения соответствующий сокращённый бюджет не
разрешается для данного диапазона конфигураций.

## Рассмотренные подходы

### Только точные низкоуровневые оптимизации

Фьюзинг статистик, переиспользование буферов и улучшение PCG имеют минимальный
риск, но не уменьшают `J * P` и количество inner-шагов. Потолок итогового
ускорения слишком низок для CG-heavy задач.

### Фиксированный сокращённый бюджет

Жёстко уменьшенные `J`, `P`, `inner_steps` и CG tolerance дают простой fast
path, но один набор параметров не может одинаково надёжно работать на разных
размерностях, функциях связи, уровнях шума и seed.

### Адаптивный каскад

Выбранный подход сочетает точный быстрый backend, вложенные уровни `J/P`,
inexact CG на ранних шагах, строгую финальную коррекцию и автоматическую
эскалацию. Он атакует оба измеренных bottleneck и сохраняет строгий fallback.

## Архитектура

Существующие `StageRegistry` и `ADPAlgorithm` сохраняются. Адаптивный режим
состоит из трёх изолированных компонентов.

### `StatisticsBudget`

Компонент выбирает текущий уровень центров и направлений и ведёт
детерминированные вложенные последовательности. Он не решает beta и не
принимает решение о качестве.

### `SolverBudget`

Компонент выбирает CG tolerance, iteration cap и необходимость строгого
корректирующего solve. Он не изменяет статистики и не знает ground truth.

### `QualityGuard`

Компонент получает telemetry статистик и solver, проверяет runtime-инварианты
и возвращает `accept`, `escalate` или `strict_fallback`. Ground-truth quality
gate применяется отдельно в калибровочном benchmark.

Общий поток outer-шага:

```text
reduced J/P -> inexact CG -> QualityGuard
                                | accept
                                | escalate -> larger J/P -> QualityGuard
                                | strict_fallback -> full J/P + strict solver
```

## Вложенные уровни статистик

Полные размеры обозначаются `J*` и `P*`.

- `L0`: `ceil(0.50 * J*)` и `ceil(0.50 * P*)`;
- `L1`: `ceil(0.75 * J*)` и `ceil(0.75 * P*)`;
- `L2`: `J*` и `P*`.

Минимальные существующие ограничения на число центров и направлений
сохраняются. Округление никогда не создаёт нулевой бюджет.

Центры и направления один раз упорядочиваются seed-зависимой перестановкой.
Каждый следующий уровень является строгим superset предыдущего. При
эскалации считаются только отсутствующие блоки `S`, `U`, `imav` и `N`;
результаты предыдущего уровня не пересчитываются.

`L1` является начальным уровнем для неизвестной конфигурации. `L0` разрешается
только для bucket из наблюдаемых runtime-признаков
`(d-range, n/d-range, active-fraction range, kernel, dtype)`, который прошёл
независимый quality gate. `L2` всегда доступен. Benchmark scenario labels не
используются policy, поскольку обычный вызов `fit` их не знает.

## Audit-набор

Десять процентов полной детерминированной последовательности центров,
не менее четырёх центров, резервируются для runtime-аудита на `L0` и `L1`.
Для них используются все направления текущего уровня. Их статистики не входят
в fit текущего сокращённого уровня, но кэшируются и переиспользуются при
эскалации до `L2`.

Audit-набор проверяет objective и направление градиента на данных, не
использованных текущим reduced fit. Он не заменяет paired quality gate, но
позволяет обнаруживать нестабильный одиночный запуск без ground truth.

Для первого кандидата audit-objective сравнивается до и после proximal-шага с
одинаковым prior. На последующих проверках используется предыдущий принятый
кандидат того же outer-шага. Рост audit-objective более чем на `0.5%`
вызывает эскалацию. `NaN`, `Inf` или невозможность вычислить audit-метрики
немедленно вызывают `strict_fallback`.

## Адаптивный inner solve

Текущая общая `tol` разделяется на независимые настройки:

- `cg_rtol_early = 1e-3`;
- `cg_rtol_stable = 1e-4`;
- `cg_rtol_final = 1e-6`;
- `inner_beta_tol = 1e-4`;
- `inner_objective_rtol = 1e-4`.

Iteration caps равны `max(20, min(100, 2 * d))` для early solve,
`max(30, min(250, 3 * d))` для stable solve и существующему
`max(50, min(500, 5 * d))` для final solve. Достижение cap не принимается как
convergence.

Ранние inner-шаги используют `cg_rtol_early`. После первого уменьшения
sign-invariant `beta_delta` ниже `1e-2` используется `cg_rtol_stable`.
Перед принятием уровня выполняется один корректирующий solve с
`cg_rtol_final`.

Inner-цикл останавливается после двух последовательных проверок, на которых
одновременно выполнены условия:

- `beta_delta <= inner_beta_tol`;
- относительное изменение objective не больше `inner_objective_rtol`;
- CG сообщает convergence;
- относительная линейная невязка не больше текущего `cg_rtol`;
- audit-objective не вырос выше допустимого порога.

Используется относительное изменение objective. Абсолютное сравнение с общей
`tol` исключается, поскольку оно зависит от масштаба задачи. Исходный hard cap
`inner_steps` сохраняется как защита от бесконечного цикла.

Эскалацию вызывают CG iteration limit, плохая невязка, осцилляция beta,
недостаточная локальная масса, неулучшающийся audit-objective или существенное
изменение beta после строгой коррекции. Осцилляцией считается отсутствие
уменьшения `beta_delta` хотя бы на `5%` в трёх последовательных проверках.
Локальная масса недостаточна, если более `5%` fitted или audit centers имеют
`N_j < min_neighbors`. Изменение после строгой коррекции существенно при
sign-invariant `beta_delta > inner_beta_tol`.

## Точное ускорение статистик

Математические формулы и явное центрирование `X_active - center` сохраняются.
Это избегает потери устойчивости из-за вычитания больших почти равных
агрегатов.

Backend должен:

- объединить вычисление weights, projected values, `S`, `imav` и `U`;
- переиспользовать рабочие буферы между центрами и chunk-вызовами;
- переиспользовать `q`, active indices и уже посчитанные блоки между уровнями;
- не создавать отдельный `np.repeat(slopes, P)` в solver path;
- выбирать center chunk из небольшого измеренного набора и кэшировать выбор
  по `(dtype, d, P, active-fraction bucket)`.

`float32` не включается глобально. Он становится допустимым только для bucket,
где измеренно быстрее `float64` и проходит тот же quality gate.

## PCG path

Рядом с текущим SciPy CG добавляется отдельный PCG stage. Он хранит residual
рекуррентно и записывает diagnostic trace без дополнительного operator
`matvec` на каждой итерации. После завершения всегда вычисляется одна точная
невязка.

В пределах outer-шага переиспользуются contiguous `u_flat`, рабочие векторы и,
если memory gate это допускает, `U**2` для Jacobi preconditioner. Warm start
между inner-шагами сохраняется.

Если число итераций систематически превышает `max(10, ceil(d / 8))`, может
быть построен детерминированный sketched SPD preconditioner. Он принимается
только если отдельно быстрее Jacobi на representative и holdout cases. Любой
preconditioner влияет только на скорость сходимости; строгая финальная
невязка остаётся обязательной.

## Параллелизм и ресурсы

В один момент используется только один уровень CPU-параллелизма:

```text
benchmark_jobs * statistics_workers * blas_threads <= physical_cpu_cores
```

Для текущей машины лимит равен шести физическим ядрам. В process-parallel
benchmark `statistics_workers=1`, а BLAS/OpenMP ограничиваются одним потоком.
В одиночном fit допускается center-parallelism, если произведение остаётся в
лимите и прогноз peak memory не превышает `1.10 * strict_baseline_peak`.

Автоматическое использование всех logical CPU запрещено. Worker pools должны
иметь явный lifecycle и закрываться после каждого ограниченного batch, чтобы
длительная работа не накапливала потоки, процессы или файловые дескрипторы.

## Ошибки и fallback

`QualityGuard` никогда не принимает кандидат при неполной telemetry.

- Ошибка reduced уровня приводит к `L2`.
- Нечисловое значение или ошибка самого guard приводит к `strict_fallback`.
- Ошибка `L2` с adaptive solver приводит к исходному SciPy CG с полными
  `J/P` и прежними tolerances.
- Ошибка strict reference path пробрасывается через существующий
  `StageExecutionError` с outer/inner context.

Fallback-причина сохраняется в результате. Молчаливое изменение режима или
возврат частичного кандидата запрещены.

## Telemetry

Каждый outer- и inner-шаг должен сохранять:

- requested и effective budget level;
- effective `J` и `P`;
- число новых и повторно использованных statistics blocks;
- текущий CG tolerance, iterations, status и final residual;
- inner stop reason;
- audit-objective и результат guard;
- escalation/fallback reason;
- раздельное время statistics, local solve, beta solve, guard и fallback.

Benchmark aggregation должна показывать доли запусков, принятых на `L0`,
`L1`, `L2` и strict fallback. Ускорение без этой информации не считается
интерпретируемым.

## Проверка

### Точные component tests

- equivalence точных statistics kernels для Epanechnikov и quartic;
- empty, sparse и dense neighborhoods;
- `float64` и `float32` dtype contracts;
- serial/parallel equivalence;
- PCG против SciPy CG на SPD, плохо обусловленных и edge-case системах;
- отсутствие дополнительного matvec при solver trace;
- детерминированность nested budgets и повторное использование блоков;
- обязательная эскалация и strict fallback для каждого failure condition.

### Performance protocol

1. Семь прогретых повторов component microbenchmarks.
2. Калибровка policy на seed `0-9`.
3. Независимая проверка на seed `10-29`.
4. Размерности `d in {5, 25, 50, 100}`, разные `n/d`, noise, correlation,
   link functions, distributions и outliers.
5. Ключевые full-profile scenarios после прохождения holdout gate.
6. Не менее 30 минут непрерывного stress-run с записью RSS, thread count,
   process count и open file descriptors.

Stress-run проходит только при ограниченном числе workers. RSS и resource
counts не должны показывать устойчивый положительный тренд после warm-up.

## Порядок внедрения

Изменения измеряются и принимаются независимо в следующем порядке:

1. зафиксировать strict reference artifacts и benchmark harness;
2. ускорить точное statistics kernel;
3. убрать лишние solver allocations и trace matvec;
4. добавить PCG и раздельные tolerances;
5. добавить relative inner stopping и строгую финальную коррекцию;
6. добавить `L1/L2`, audit и fallback;
7. калибровать и при необходимости разрешить `L0`;
8. провести holdout, full и long-run verification;
9. рассматривать compiled CPU kernel только если принятые изменения не
   достигают performance gates.

Каждый слой сохраняется только после самостоятельного прохождения quality и
performance gates. Последующий слой не может скрывать регрессию предыдущего.

## Не-цели первого цикла реализации

- изменение математической ADP objective;
- изменение генератора benchmark-данных;
- обязательная GPU-зависимость;
- оптимизация local solver, который не является измеренным bottleneck;
- автоматическая загрузка всех CPU без resource budget;
- удаление strict reference mode.

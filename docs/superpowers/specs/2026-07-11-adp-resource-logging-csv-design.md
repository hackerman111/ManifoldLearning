# ADP Resource Logging and CSV Experiment Storage Design

## Goal

Добавить честное измерение времени и RSS-памяти для самого алгоритма ADP и
полного экспериментального запуска, а также заменить JSON-артефакты серий
экспериментов нормализованными CSV-таблицами.

## Measurement contract

### Algorithm window

Окно алгоритма начинается при входе в `ADPAlgorithm.fit()` до преобразования
входных массивов и заканчивается после формирования `ADPResult` и очистки
временных backend-ресурсов. Результат содержит:

- `algorithm_time_sec`;
- `algorithm_rss_start_mib`;
- `algorithm_rss_min_mib`;
- `algorithm_rss_mean_mib`;
- `algorithm_rss_max_mib`;
- `algorithm_rss_peak_delta_mib`;
- `algorithm_memory_samples`.

Существующий `result.timings["total"]` сохраняется для обратной совместимости.
Новое поле `algorithm_time_sec` является явным wall-clock временем полного
вызова `fit()` и не подменяется суммой времен отдельных этапов.

### Full-run window

Окно полного запуска начинается перед генерацией данных и созданием модели. Оно
включает `fit()`, расчет метрик и запись основных строк результата в CSV. Запись
самой строки с ресурсными метриками выполняется после остановки монитора и не
может входить в собственное измерение. Для каждого job сохраняются:

- `full_run_time_sec`;
- `full_run_rss_start_mib`;
- `full_run_rss_min_mib`;
- `full_run_rss_mean_mib`;
- `full_run_rss_max_mib`;
- `full_run_rss_peak_delta_mib`;
- `full_run_memory_samples`;
- `result_persist_time_sec`.

Если job завершается ошибкой, монитор все равно останавливается, а доступные
метрики и сообщение об ошибке сохраняются в таблице запусков.

### RSS sampler

Общий монитор находится в `adp/common/resource_monitor.py`. Он снимает текущий
RSS процесса через `psutil`, когда библиотека доступна. На Linux без `psutil`
используется `/proc/self/statm`; крайний переносимый fallback использует
`resource.getrusage` и явно помечает источник. Монитор всегда делает начальный
и финальный снимок, поэтому короткие операции имеют не менее двух измерений.

Измерения агрегируются онлайн: монитор хранит только количество, сумму,
минимум и максимум, а не весь временной ряд. Период по умолчанию равен 10 мс.
RSS измеряется в MiB и включает NumPy-буферы, в отличие от `tracemalloc`.

Вложенные мониторы допустимы: полный запуск и `fit()` имеют независимые окна и
отдельные sampling threads. В параллельном режиме каждый worker измеряет только
свой процесс.

## Result model and public diagnostics

`ADPResult` получает словарь `resource_usage` с алгоритмическими полями. Метод
`model.summary()` возвращает его отдельным разделом, не смешивая с накопленными
`stage_timings`. Последнее измерение также хранится на модели, чтобы runner мог
записать ресурсные данные неуспешного `fit()`.

Измерение включено по умолчанию. Оно не меняет математический результат и не
должно зависеть от backend. Старые поля времени и памяти в benchmark/stress CSV
сохраняются как совместимые aliases на новые явные поля там, где это требуется
существующими потребителями.

## CSV storage architecture

Для серий используется общий writer в `adp/common/experiment_log.py`. Он:

- фиксирует порядок и схему колонок;
- добавляет `schema_version` и стабильный `run_id`;
- пишет строки потоково через стандартный модуль `csv`;
- проверяет совпадение заголовка при append;
- объединяет worker-specific shards без загрузки всех строк в память;
- удаляет временные shards только после успешного объединения.

В параллельном режиме каждый процесс пишет в собственный shard, определяемый
PID. Это исключает конкурентную запись в один файл и позволяет включить
сохранение основного результата в full-run window. Итоговый порядок строк
определяется ключом `run_id`, `outer_k`; если для построения отчета необходима
сортировка, она выполняется отдельно после объединения и не меняет run log.

## Tables

Каждая серия создает следующие таблицы с общим префиксом:

### `<prefix>_runs.csv`

Одна строка на job:

- `schema_version`, `series_id`, `run_id`;
- experiment/scenario/method/seed и размеры задачи;
- `data_seed`, `fit_seed`;
- статус, тип ошибки и текст ошибки;
- алгоритмическое время и RSS-метрики;
- end-to-end время, RSS-метрики и время сохранения;
- число сохраненных iteration rows.

### `<prefix>_iterations.csv`

Одна строка на outer-итерацию. Сохраняет текущий confirmatory records contract:
`h_k`, `rho_k`, локальную массу, качество направления, objective и признаки
успеха. `run_id` связывает строку с `runs.csv`.

### `<prefix>_initial_parameters.csv`

Одна строка на job. Вложенные `scenario`, `ADPConfig` и описание `beta0`
разворачиваются в колонки с префиксами `scenario_`, `config_` и `beta0_`.
Массивы и словари не сериализуются как JSON внутри ячеек.

### `<prefix>_summary.csv` and `<prefix>_final_success.csv`

Сохраняют существующие агрегаты, дополняясь средними, медианами и квантилями
алгоритмического и полного времени и RSS-памяти.

### `<prefix>_series.csv`

Одна строка с `schema_version`, `series_id`, временем начала/завершения,
числом requested/completed/failed jobs, requested/actual workers, общим временем
и ошибкой parallel fallback. Параметры серии разворачиваются в отдельные
колонки `config_*`.

### `<prefix>_artifacts.csv`

По одной строке на артефакт: `artifact_type`, `name`, `path`. Таблица заменяет
JSON `outputs`/`plots` и включает пути к CSV и PNG.

## Migration scope

Общий ресурсный монитор применяется в core ADP, benchmark и stress runners.
CSV-схема внедряется в поддерживаемые серии:

- `experiments/adp_confirmatory_common.py` и entrypoints экспериментов 4/5/6;
- `adp/evaluation/stress.py`;
- `experiments/benchmark_numpy_statistics.py`;
- общий benchmark report в `adp/evaluation`.

Старые JSON manifest/initial-parameters файлы этих путей больше не создаются.
Исследовательские legacy-скрипты `adp_h0_research*.py` остаются вне миграции:
они имеют отдельный формат статистических extra tests и не вызываются через
поддерживаемые `run_benchmarks.py` или confirmatory entrypoints.

## Failure and recovery behavior

- Ошибка одного job не останавливает последовательную серию и сохраняется в
  `runs.csv`.
- Ошибка создания или записи итогового CSV считается ошибкой серии и не
  маскируется.
- При недоступности process pool сохраняется существующий sequential fallback;
  причина записывается в `series.csv`.
- Неполные shards сохраняются при аварийном завершении, чтобы результаты можно
  было восстановить; после успешного merge временный каталог удаляется.
- Пустая серия все равно создает CSV с заголовком и строку `series.csv`.

## Testing

Тесты следуют red-green циклу и проверяют:

1. Монитор возвращает конечные min/mean/max, правильный порядок значений,
   минимум два samples и положительное время.
2. Выделение NumPy-массива отражается в RSS max/peak delta на поддерживаемом
   RSS backend без жесткого требования к точному значению.
3. Успешный и неуспешный `fit()` оставляют ресурсные метрики.
4. End-to-end job содержит оба независимых набора метрик и включает запись
   iteration rows.
5. Последовательная и параллельная серии создают одинаковые CSV-схемы.
6. Начальные параметры полностью разворачиваются в CSV без JSON-ячеек.
7. Старые JSON-файлы поддерживаемых runners не создаются.
8. Summary агрегирует новые время и память.
9. Полный test suite и `git diff --check` проходят после миграции.

## Compatibility and documentation

README получает описание двух окон измерения, единиц RSS и перечень новых CSV.
Изменение формата JSON → CSV является намеренно несовместимым для файловых
потребителей; имена возвращаемых ключей runner API обновляются на `runs`,
`iterations`, `series` и `artifacts`. Математический API `ADP.create(...).fit()`
остается совместимым.
